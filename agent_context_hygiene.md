## 一、什么是上下文污染

在多 Agent 系统中，每个 Agent 的上下文包括：系统提示、任务书、其他 Agent 输出、工具返回、协调者转述。

**污染** = 上述内容中，有不该进入当前 Agent 决策边界的信息，却被当作事实使用。

### 常见污染类型（本项目真实踩过的坑）

| 类型 | 本项目实例 |
|------|-----------|
| **代录污染** | Grok 单人写 5/5 APPROVED 共识，其他 Agent 从未落盘（v6.3.1） |
| **摘要失真** | 协调者把 NEEDS_CHANGES 概括成"基本同意" |
| **时序污染** | R2 辩论读 R1 文件，但基线版本已变（v6.3.1 vs v6.3.2） |
| **权限污染** | Agent A 的假设被写入 task，Agent B 当成已验证事实 |
| **状态幻觉** | "已 push""E2E 全过"写在共识里，但 git/测试并未执行 |
| **角色越界** | 协调者代投、代写 review、代合并票（Grok v6.3.1） |
| **环境边界** | Agent 读取 /mnt/f/ 权限拒绝，任务文件不在可写目录 |
| **跨轮语义漂移** | culture_weight vs regional_modifiers 不一致 |

## 二、核心判断标准

```
当前 Agent 的 vote / 结论，是否只能由它自己读过的、可溯源的原始文件支撑？
若不能 → 就是污染架构，不是"模型不够好"。
```

---

## 三、信息论约束

1. **来源可追溯（provenance）**：每条结论必须对应 Agent + 文件 + 时间戳
2. **写读分离（write-once, read-many）**：Agent 只读他人已落盘文件，不读协调者转述

## 四、防污染三件套

| 机制 | 作用 | v2 协议实现 |
|------|------|-----------|
| **文件即投票** | 无 `*_review.md` = 无票，不是默认同意 | ✅ 已执行 |
| **轮次隔离** | R1 独立落盘；R2 才允许读他人文件 | ✅ 已执行 |
| **协调者只读合并** | Grok 禁止生成他人 review 正文 | ✅ 已执行 |

## 五、5 条硬标准（判断多 Agent 回答质量）

缺 2 条以上 → 回答偏空泛，不可信。

### 标准 A：是否区分「协调」与「决策」

- ✅ 好：协调者只调度、合并、跑 E2E，不代投、不代写他人 review
- ❌ 差："让 Grok 汇总大家意见然后推进"——没说明汇总是否基于真实文件

### 标准 B：是否定义上下文边界

- ✅ 好：每个 Agent 的输入白名单（例：只读 `*_scope_r1.md`，不读 consensus 草稿）
- ❌ 差："大家共享一个 thread"——典型污染温床

### 标准 C：是否有「缺票 / 超时」语义

- ✅ 好：TIMEOUT = ABSTAIN，分母不含缺席者，禁止默认 APPROVED
- ❌ 差："3 个 Agent 同意就够了"——没定义第 4、5 个去哪了

### 标准 D：是否区分轮次（R1 独立 → R2 交叉 → R3 锁版）

- ✅ 好：R1 禁止读他人输出；R2 必须引用他人原文并标注同意/争议
- ❌ 差：一轮 prompt 里塞 5 份 review 让模型"综合"——那是单模型假装多 Agent

### 标准 E：是否有可验证工件（artifact）

- ✅ 好：`20260623_mimo_v64_lock_r2.md` + `LOCK_VOTE: APPROVED` + E2E 19/19
- ❌ 差："经讨论一致认为可以锁版"——无文件、无 commit、无测试

## 六、有效追问模板

当有人提多 Agent 方案时，按顺序拷问：

| # | 追问 | 测什么 |
|---|------|--------|
| 1 | 每个 Agent 的输入文件清单？明确禁止读什么？ | 边界设计 |
| 2 | 协调者写 consensus 时，合并规则？缺 1 票怎么算？ | 投票语义 |
| 3 | 若 Hermes CLI 超时 90s，系统行为是 ABSTAIN 还是 Grok 代写？ | 失败模式 |
| 4 | R1 和 R2 的上下文是否隔离？R2 如何证明读过他人原文而非摘要？ | 交叉辩论真实性 |
| 5 | 锁版门禁绑定什么可执行证据？git SHA、回归脚本、还是聊天里说 PASS？ | 可审计性 |
| 6 | 同一任务并行跑 3 次，consensus 文件是否 byte-level 可复现？ | 确定性 vs 幻觉 |

任一回答含糊其辞 → 方案在架构层不合格。

## 八、一句话总结

```
多 Agent 的价值 = 独立上下文下的独立判断；
多 Agent 的失败 = 共享了被协调者污染过的上下文，却假装有分歧与共识。
```

判断任何 AI 回答，先问三句：

1. **票从哪来？**（文件 / 聊天 / 协调者脑补）
2. **上下文谁看过？**（白名单 / 全 thread）
3. **锁版靠什么？**（SHA + 测试 / 口头 PASS）

三句都有清晰答案 → 架构认知到位。任一句说不清 → 先修架构，再调模型。
